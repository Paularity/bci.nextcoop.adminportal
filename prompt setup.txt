# NextCoop — System Admin Portal — Full Project Setup Prompt

Use this prompt with Claude Code (or paste into a new chat) to scaffold the System Admin Portal for this sprint. It supersedes the earlier "mock SQLite, frontend-first" draft — this version adds the REST API layer required by PBI 24729 and maps every requirement to the five sprint PBIs. Read this whole document before writing any code; it defines the full contract the implementation must satisfy.

---

## 0. Non-negotiable constraints

- Single **SQLite** database file (`prisma/dev.db`), managed by **Prisma**, `provider = "sqlite"`. No control-DB/per-tenant-DB split — that architecture is deferred (see §8).
- The **web application must consume tenant data through the internal REST API** (`app/api/tenants/...`) — **not** by calling Prisma/the repository directly from Server Components. This is a hard requirement from PBI 24729 (AC-API-*), and it's a deliberate shift from the earlier draft.
- Server Components fetch from the internal API using server-side `fetch`. Server Actions (triggered by native `form action={...}`) also call the internal API rather than touching Prisma directly — Server Actions are HTTP clients to your own API, not a bypass of it.
- Every mutation still goes through Zod validation as the trust boundary — once client-side (via the API route handler receiving the request) and once more defensively if a Server Action pre-validates before calling the API.
- Full SSR philosophy from the earlier draft still holds: URL `searchParams` drive filtering/sort/pagination, native `form` elements drive mutations. The two justified client-side exceptions are the sidebar toggle and the tenant data table (TanStack Table + shadcn, see §1) — both are UI-rendering components, not data-fetching ones; the data itself still arrives via server-side fetch to the internal API and is passed down as props.
- Every PBI below gets its own branch off `main`: `feature/{PBI-ID}-short-desc`.

**Package rationale (so the AI implementing this doesn't second-guess the choices):**
- `next-auth@beta` (Auth.js v5) — Credentials provider against the local `User` table, JWT session strategy. Chosen over hand-rolled cookies because AC-LGN-06 requires a "secure session" and AC-LGN-05 requires role-based authorization at the auth layer — Auth.js gives both via `callbacks.jwt` / `callbacks.session` without custom cookie-signing code.
- `bcryptjs` over `bcrypt` — pure JS, no native build step, fine for a mock/dev-scale SQLite setup.
- `shadcn/ui` — copies component source into the repo (not a runtime dependency), Tailwind-based, matches the "no Kendo UI" constraint from the base architecture doc.
- `@tanstack/react-table` — headless table logic (column defs, sorting/pagination state machinery) paired with shadcn's `table` primitive for markup. Used in **manual/controlled mode** (`manualSorting`, `manualPagination`, `manualFiltering: true`) — TanStack Table does not own the data or trigger its own fetches; it only renders columns and reflects sort/page state that still lives in the URL. This supersedes the earlier "no client-side table library" constraint specifically for tables — everything else (search form, mutation forms) stays server-driven per §0.
- `zustand` — sidebar collapse state only. Nothing tenant-data-related ever touches Zustand; that's exclusively Server Component + `searchParams` territory.

## 2. Database schema (Prisma / SQLite)

```prisma
// prisma/schema.prisma
datasource db {
  provider = "sqlite"
  url      = env("DATABASE_URL") // file:./dev.db
}

generator client {
  provider = "prisma-client-js"
}

enum UserRole {
  SYSTEM_ADMIN
  TENANT_ADMIN
}

enum TenantStatus {
  ACTIVE
  INACTIVE
}

model User {
  id           String   @id @default(cuid())
  username     String   @unique
  email        String   @unique
  passwordHash String
  firstName    String
  lastName     String
  mobileNumber String?
  role         UserRole
  tenantId     String?             // set for TENANT_ADMIN users, null for SYSTEM_ADMIN
  tenant       Tenant?  @relation(fields: [tenantId], references: [id])
  createdAt    DateTime @default(now())
  updatedAt    DateTime @updatedAt
}

model Tenant {
  id                String       @id @default(cuid())
  tenantCode        String       @unique
  cooperativeName   String
  cooperativeAddress String
  status            TenantStatus @default(ACTIVE)
  administrators    User[]
  createdAt         DateTime     @default(now())
  updatedAt         DateTime     @updatedAt
}

model AuditLog {
  id          String   @id @default(cuid())
  entityType  String   // "Tenant"
  entityId    String
  action      String   // "CREATE" | "UPDATE" | "ACTIVATE" | "DEACTIVATE" | "DELETE"
  performedBy String   // User.id of the acting System Admin
  changes     String?  // JSON string of before/after diff, nullable for CREATE
  createdAt   DateTime @default(now())
}
```

Seed script (`prisma/seed.ts`) must create:
- One `SYSTEM_ADMIN` user (for login testing — document the mock credentials in the README, do not hardcode them silently with no reference).
- Two or three mock `Tenant` rows, each with one `TENANT_ADMIN` user, to populate the dashboard grid and exercise search/pagination immediately.

## 4. Internal REST API contract (PBI 24729)

| Method | Route | Purpose | Auth | Success | Errors |
|---|---|---|---|---|---|
| POST | `/api/tenants` | Create tenant + tenant admin in one transaction | SYSTEM_ADMIN session required | `201` + created tenant | `400` validation, `409` duplicate tenant code / username, `401`/`403` |
| GET | `/api/tenants` | List tenants, supports `?q=&status=&page=&sort=&order=` | SYSTEM_ADMIN session required | `200` + `{ data, total, page }` | `401`/`403` |
| GET | `/api/tenants/[id]` | Get one tenant + its administrator | SYSTEM_ADMIN session required | `200` + tenant | `404`, `401`/`403` |
| PUT | `/api/tenants/[id]` | Update tenant/administrator fields | SYSTEM_ADMIN session required | `200` + updated tenant, audit log written | `400`, `404`, `401`/`403` |
| DELETE | `/api/tenants/[id]` | Delete or deactivate per business rule | SYSTEM_ADMIN session required | `200`/`204`, audit log written | `404`, `401`/`403` |
| POST | `/api/tenants/[id]/activate` | Set status ACTIVE | SYSTEM_ADMIN session required | `200`, audit log written | `404`, `401`/`403` |
| POST | `/api/tenants/[id]/deactivate` | Set status INACTIVE | SYSTEM_ADMIN session required | `200`, audit log written | `404`, `401`/`403` |

- **AC-DTL-06 / AC-DSH-07 business rule to implement explicitly:** decide and hardcode the rule — recommend *soft delete* (set an internal flag or reuse `INACTIVE` + a `deletedAt` timestamp) rather than a hard `DELETE FROM`, since tenants have dependent `User` rows and an audit trail requirement. State this decision in the README; don't leave it ambiguous in code.
- Every route handler: `safeParse` the request body with the shared Zod schema before touching Prisma, return `400` with field-level messages on failure (AC-API-06, AC-CRT-02).
- Every route handler checks the session and role before doing anything else; return `401` if no session, `403` if session exists but role isn't `SYSTEM_ADMIN` (AC-API-07, AC-LGN-05).

## 5. PBI-to-implementation map

### 24475 — Login (`feature/24475-login`)
- `(auth)/login/page.tsx`: native form, `username`/`password` fields, submit button, error message region (AC-LGN-01, AC-LGN-04).
- Server Action → Auth.js `signIn("credentials", ...)`. On success, redirect to `/tenants` (AC-LGN-02). On failure, redisplay form with an authentication error, don't leak whether it was the username or password that was wrong (AC-LGN-03).
- Auth.js `callbacks.jwt`/`session` embeds `role`. `middleware.ts` rejects any session where `role !== "SYSTEM_ADMIN"` on `(admin)/*` routes (AC-LGN-05).
- Session strategy: JWT, `httpOnly` secure cookie (AC-LGN-06).

### 24726 — Dashboard (`feature/24726-tenant-dashboard`)
- `(admin)/tenants/page.tsx`, Server Component, is the landing page post-login (AC-DSH-01).
- Fetches from `GET /api/tenants?...searchParams` — not Prisma directly. Passes the resulting `data`/`total`/`page` down as props into `<TenantTable>`.
- `tenant-table.tsx` (Client Component) wraps `useReactTable` with `manualSorting: true`, `manualPagination: true`, `manualFiltering: true` — it renders sort-arrow buttons and pagination controls, but clicking them does `router.push` with updated `searchParams` rather than mutating any local table state. The actual re-fetch happens because Next.js re-runs the Server Component on navigation, same as the plain-links approach from the earlier draft — TanStack Table is purely the rendering/interaction layer on top of that.
- `tenant-columns.tsx` defines columns: Tenant Code, Cooperative Name, Tenant Admin (first+last name), Email, Status (via `status-badge.tsx`), Created Date (AC-DSH-02).
- `tenant-search.tsx` (Server Component): `form method="GET"`, `name="q"` (AC-DSH-03).
- "Create Tenant" button linking to `/tenants/new` (AC-DSH-04).
- Row actions column: View → `/tenants/[id]`, Edit → `/tenants/[id]/edit`, Delete → confirm route (AC-DSH-05/06/07).
- `empty-state.tsx` renders when `data.length === 0`, in place of the table (AC-DSH-08).

### 24727 — Create Tenant (`feature/24727-create-tenant`)
- `(admin)/tenants/new/page.tsx`: single form, three field groups — Cooperative (name, address), Administrator (first name, last name, mobile, email, role), Account (tenant code, username) (AC-CRT-01).
- Zod schema requires all mandatory fields; missing fields render inline validation messages on redisplay, not just a toast (AC-CRT-02).
- Server Action posts to `POST /api/tenants`. The route handler wraps tenant creation + administrator `User` creation in a single `prisma.$transaction([...])` — both succeed or both roll back (AC-CRT-03).
- Uniqueness checks (`tenantCode`, `username`) run inside the same transaction/route handler before insert; on conflict return `409` and surface a field-specific error ("Tenant code already exists" / "Username already exists") rather than a generic failure (AC-CRT-04, AC-CRT-05).
- On success: success message (e.g. via a redirect query param rendered as a toast/banner on `/tenants`) and redirect to the dashboard (AC-CRT-06).
- Cancel is a plain link back to `/tenants` — no form submission, nothing written (AC-CRT-07).

### 24728 — Tenant Details (`feature/24728-tenant-details`)
- `(admin)/tenants/[tenantId]/page.tsx`: fetches `GET /api/tenants/[id]`, renders tenant + administrator info and current status badge (AC-DTL-01, AC-DTL-02).
- `edit/page.tsx`: prefilled form, submits to `PUT /api/tenants/[id]` (AC-DTL-03).
- `activate/page.tsx` / `deactivate/page.tsx`: confirmation routes posting to the respective API endpoints (AC-DTL-04, AC-DTL-05).
- `delete/page.tsx`: confirmation route, applies the soft-delete/deactivate rule decided in §4 (AC-DTL-06).
- Every mutating route handler (`PUT`, `DELETE`, `activate`, `deactivate`) calls `lib/audit.ts` → writes an `AuditLog` row with the acting System Admin's `User.id`, before/after diff for updates (AC-DTL-07).

### 24729 — Tenants API (`feature/24729-tenants-api`)
- Implement the full contract in §4. This PBI is the dependency for 24726/24727/24728 — sequence it first, or stub the route handlers with mock in-memory responses on day one so the frontend branches aren't blocked while the real Prisma-backed logic lands.
- Standard response shape for all endpoints: `{ data: ... }` on success, `{ error: { message, fields? } }` on failure, matched to the correct HTTP status code (AC-API-08).

## 6. Guides for implementation

**Validation:** one Zod schema per resource action (`createTenantSchema`, `updateTenantSchema`, `loginSchema`) in `lib/schemas/`. Route handlers and any Server Action that pre-checks both import the same schema — never redefine field rules in two places.

**Auth guard pattern:** `middleware.ts` handles route-level redirect-to-login for unauthenticated users hitting `(admin)/*`. Individual API route handlers still re-check `session.role === "SYSTEM_ADMIN"` themselves — middleware is a UX convenience, not the security boundary for the API.

**Audit logging:** every state-changing API route handler calls `writeAuditLog` after a successful Prisma write, inside the same function, not as an afterthought in the UI layer. If the write itself fails, no audit row should be written for that action.

**Error surfacing convention:** validation and conflict errors return structured JSON (`{ error: { message, fields } }`) from the API; the Server Action or Server Component reading the API response maps that into the redirect query param / inline form error the page renders. Don't let raw API error bodies leak to the UI unformatted.

**Response payload consistency:** never return `passwordHash` in any API response, including nested administrator objects inside a tenant payload.
